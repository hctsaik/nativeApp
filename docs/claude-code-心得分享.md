# 用 Claude Code 開發的習慣與心得分享

> 整理自一段長期、真實的協作開發經驗（CIM Hybrid Edge Platform：Electron + React + Python sidecar 的混合平台，跨數十個對話、上千個測試）。
> 這份文件不是「Claude Code 功能列表」，而是**一個工程師實際養成的習慣、踩過的坑、和值得抄的招**。
> 文中所有評分軌跡、踩坑案例、規則，都來自這個 repo 的設計文件與記憶檔——**不是示範用的假資料**。

---

## 一句話心法

> **不要把 Claude Code 當「會打字的補全工具」，把它當成「一個可以被你『程式化』的協作夥伴」。**

差別在於：補全工具是被動的；協作夥伴需要你**給它記憶、給它流程、給它驗收標準、給它防呆規則**。
下面 10 個 tip，本質上都在做這四件事。

## Tip 1 — 記憶系統：跨 session 的「第二大腦」

最大的痛點是：每開一個新對話，AI 就失憶。我的解法是把**值得跨 session 的事實**寫成結構化的記憶檔。

**做法**
- 每則記憶一個檔案，一個事實，帶 frontmatter 分類：
  - `user`（我是誰、偏好）／`feedback`（我給過的工作指引＋為什麼）／`project`（進行中的工作、決策）／`reference`（外部連結）
- 一個 `MEMORY.md` 當索引，每則記憶一行，開新對話時自動載入。
- 記憶之間用 `[[name]]` 互相連結，像個人 wiki。

**配套的兩個自製指令（這是關鍵）**
- `/checkpoint`：對話結束前，掃描這次完成了什麼、把 `tasks.md` 的 `- [ ]` 改成 `- [x]`、把斷點用固定模板（本次完成／下一步／相關檔案／重要決策）寫進 `current_focus.md`、更新 `MEMORY.md` 索引。
- `/resume`：新對話開始時，讀斷點 + 掃所有未完成 `tasks.md`，輸出一份「開發進度簡報」，結尾固定問我「準備好從『〔下一步〕』繼續了，要開始嗎？」

> 巧思：checkpoint 與 resume **共用同一個 `current_focus.md`**，而且各段標題一一對齊（一個寫「本次完成」、另一個讀「本次完成」）——等於把人腦的「上次做到哪」固化成一個機器可讀的斷點檔。

**為什麼值得抄**
> 「記憶」不是讓 AI 記住程式碼結構（那 git 和 codebase 本來就有），
> 而是記住**從程式碼看不出來的東西**：為什麼這樣決定、我糾正過它什麼、現在做到哪、下一步是什麼。

**一個重要紀律**：記憶是「當時的觀察」，不是即時狀態。引用到某個 `file:line` 前，**先驗證它還存在**——程式碼會變，記憶會過期。（我的記憶檔載入時甚至會自帶一行提醒「這則記憶 N 天前寫的，引用前請對照現在的程式碼」。）

## Tip 2 — 把 `CLAUDE.md` 當成「上手文件 + 防雷手冊」

`CLAUDE.md` 是每次對話都會載入的最高優先指令。我把它寫成一份**真正的團隊文件**，而不是隨手記幾行。

我的 `CLAUDE.md` 裡有幾個高價值區塊：
- **啟動方式 / 首次設定**：連「解壓 zip 或全新 clone 後要做什麼」都寫清楚，可 E2E 跑通。
- **架構關鍵點**：啟動鏈、事實來源（哪個是權威、哪個是衍生快取）、環境變數由誰注入。
- **「常見錯誤與處理」表格**：錯誤訊息 → 原因 → 解法。AI 一撞到就知道怎麼修。
- **「架構地雷」清單**：哪些地方改了會連帶炸。

**這些規則每一條都是踩坑後回頭寫的**，舉三個最有故事性的：

- **`ELECTRON_RUN_AS_NODE=1` 殘留 → Electron app 啟動印個 Node 版本就退出**。元凶是 **Claude Code 自己設了這個環境變數**——工具反過來污染了它要開發的 app。固化成規則：移除該變數，或用 `launch-electron.js` workaround。
- **WDAC 封鎖未簽章 exe**：先是 `xanylabeling.exe`（uv trampoline）被企業 Application Control 擋，規則寫死「必須用 `py -3.11 -c "...main()"` 啟動，**不要改回直接執行 exe**」；後來**同一個坑換個受害者又踩一次**——Vite 的 `esbuild.exe` 也未簽章被擋，導致 `start-dev` 整個起不來。把「未簽章 exe 會被 WDAC 咬」寫進知識庫，下次就不必重新 debug 半天。
- **新增 postMessage 類型要同步改 3 個地方**（`MessageTypes` + 測試 + 協定白名單），否則 `isProtocolMessage` 會**靜默過濾**掉新訊息——程式看起來對，但 portal 收不到。

**為什麼值得抄**
> 每一條「常見錯誤」和「架構地雷」，都是某次踩坑後**回頭寫進去的**。
> 等於把「血淚教訓」固化成 AI 下次一定看得到的規則，而不是期待它自己想起來。

**反例提醒**：別把 codebase 自己看得出來的東西（檔案結構、函式簽名）抄進 `CLAUDE.md`——那只是雜訊。只寫「看不出來但很重要」的。

## Tip 3 — 把重複流程固化成 Skills / Slash Commands

凡是「我會講第二次」的流程，我就把它變成一個自製指令或 skill。我累積的例子：

| 指令 / Skill | 做什麼 |
|---|---|
| `/checkpoint`、`/resume` | session 接續（見 Tip 1） |
| `/ux-test`、`e2e-test` | persona + MCP 的真實 UX 驗收（見 Tip 6） |
| `/package-source`、`/package-build`、`/publish-bundle` | 一鍵打包原始碼 zip / 可攜安裝包 / 全家桶 |
| `/new-cv-module`、`/new-split-tool` | 新工具骨架（先問固定參數 → 生成固定結構 → 把易踩的坑寫成必讀規則 → 末端強制跑 pytest） |
| `unet-method` skill + `/user /po /architect /pm /pg` | 五角色開發流程（見 Tip 5） |

**為什麼值得抄**
> Slash command 本質上就是「**把一段你滿意的 prompt 存起來，給它取個名字**」。
> 你不必每次重講「打包時要排除哪些目錄、檢查哪些東西」——講一次，存成 `/package-source`，之後一個字搞定。

**判準**：流程要寫超過 3 行、而且你會用第二次 → 就值得固化成指令。
**放哪**：跨專案通用的（角色指令、ux-test 骨架）放 `~/.claude/`，專案專屬的（打包、建模組骨架）放 repo 的 `.claude/commands/`。

## Tip 4 — 多 Agent 評分收斂法：用「一群評審」做關鍵決策

遇到「這個設計到底好不好 / 這個平台對目標使用者夠不夠易用」這種**沒有單一正確答案**的問題，我不靠單次回答，而是派一群評審來打分、多輪迭代。

**先學會下一個能讓 AI 自主長跑的目標 prompt。** 我真實下過的 `/goal`（節錄）：

> 「用 Multi agent 定義十種情境（半導體晶圓製造業的軟體工程師如何用這平台、自己開發系統做上架），透過 Multi agent 打分數，**如果沒有達到平均 95 分之前不要停下來，而是再產生十個情境反覆重新驗證**……在還沒完成之前請不要停下來，你認為有幫助之前請不要叫我。」

這一段把三件事一次講清楚：**目標、量化門檻、自主迭代的停止準則**。

**方法三步**
1. 訂一個明確的 **rubric**。例如「每情境 0–100，從零到上線無摩擦＝高分；需手寫大量樣板＝扣分；平台根本沒這條路、要改核心＝大幅扣分」，並拆成可解釋的維度（可達成度／技術門檻／步驟摩擦／可發現性／防呆／可維護）。
2. 派**多個獨立 agent**，各自用情境 / persona 去打分。情境要**具體到有畫面**——我用過的真實情境像：`T1 機台保養點檢表單`（零 Streamlit code 上線）、`T2 廠內量測 GUI`（啟動桌面程式、作業完自動回收結果檔，對標 Label tool 的完整迴圈）、`T7 接 SECS/GEM 非-REST 設備`（要寫 connector 並讓平台自動選用）。
3. 多輪迭代：每輪找**最高槓桿的缺口**去修，再重評，記錄**誠實的分數軌跡**。

**真實軌跡（直接從設計文件摘出來，含掉分）**
- 「平台是否易用的 no-code 平台」（跑了 **15 輪**）：`62 → 66 → 64 → 64 → 74.5 → 79.6 → 84.4 → 87.9 → 82.5 → 86.1 → 89.1 → 85.0 → 85.0 → 89.3 → 89.2`。
- 「工程師易開發 + 上架工具」（12 輪）：`43 → 65 → 74 → 78 → (R4 ≈74 回落) → 85 → 91 → 91.5 → 89 → 89.5 → 94.2 → 94.4 → 94.4`。
- 「LV BDD E2E 套件品質」：`88.4 → 93.8 → 95.4`（**這個真的衝過 95**）。

**三個最值得講的細節**

- **掉分是好事，代表沒灌水。** no-code 第 9 輪從 87.9 掉到 82.5，不是退步，是「這輪刻意公平取樣、納入更多硬骨頭情境，揭露兩個領域內真缺口」。第 12 輪標題直接寫「**誠實回落**」——評估官實地查證後發現上一輪太樂觀，如實計入。
- **fake-green 會讓分數虛高，被抓到要止血。** 自建工具第 4 輪不升反降，因為重評揭露「半套」：平行大量 `Edit` 有幾筆**靜默失敗**、兩個守門測試**其實是紅燈卻被當成完成**。這條教訓後來變成 Tip 7。
- **誠實停在 94.4，而不是硬湊 95。** 兩位評分官的結論是：所有 <95 的扣分 **100% 落在使用者明文排除的範疇**（正式 SSO/IdP、視覺化拖拉式 builder、生產級打磨），**沒有任何一分是因為「走不通／空殼／名實不符」**。最後那 0.6 分要靠投入使用者明確不要的工程才補得起來，邊際效益遞減——所以定案、轉維護。

**為什麼值得抄**
> 單一 agent 容易**自我感覺良好**。多個獨立評審 + 明確 rubric + 多輪，會逼出「看起來對、用起來爛」的問題。
> 而且要接受**收斂常是停在高原而非單調逼近滿分**：每輪補上真缺口，但新功能又暴露更小的新摩擦，所以均分在 ~89 收斂。**誠實記錄這條軌跡，比假裝滿分有價值得多。**

**怎麼記錄**：每輪固定欄位「討論項目 / 共識 / 爭議 / 後續方向」。多方辯論時我會用真實的角色配置——例如架構決策用「四視角 → 收斂成結構派 vs 務實派兩陣營」、模組拆分用「五視角各打一面向分」、工具評估用「雙評分官甲乙交叉複評」。
**搭配工具**：這類大規模 fan-out，Claude Code 的多 agent / workflow 能力可以並行跑——背景 agent 跑情境分析、前景 agent 跑 MCP 操作、再綜合三方報告。

## Tip 5 — U-Net 五角色工作流：規格驅動 + TDD + 機器強制的階段閘門

開發稍有規模的東西時，我用一套自創的「U-Net Agent Workflow」。本質**不是神經網路**，是用一個好記的形狀把三件紀律綁在一起：**Spec-Driven + TDD + Phase Gating**。

在**單一終端機**裡，靠「角色切換 + 資料夾隔離 + 階段閘門」模擬一支多角色團隊：

| 指令 | 角色 | 只能動的資料夾 | 鐵則 |
|---|---|---|---|
| `/user` | 終端使用者 | `1_user_needs/` | 只描述「要什麼、為什麼痛」，**嚴禁碰技術**；成功判準要可觀察（「點下去 3 秒內看到結果」勝過「很快」） |
| `/po` | 產品負責人 | `2_PO_PRD/` + `ROADMAP.md` | 唯一負責把 feature 拆成 module 的**粒度轉換點**；附拆模組自檢三問 |
| `/architect` | 架構師 | `3_Architect_Design/` | 寫 I/O 契約與 AC，**嚴禁寫實作**；AC 要釘死數值（`f(75,1.0,30,150)==(45,105)`），**禁止「element 存在/非空」這類代理式 AC** |
| `/pm` | 專案經理 | `4_PM_Feedback/` | 先寫**會紅的**驗收測試（test-first）；**不准只逐字抄 AC**，每模組至少加一條自行推導的不變量/metamorphic 測試 |
| `/pg` | 程式設計師 | `5_PG_Develop/` | 寫實作、自動跑到全綠；**不准改測試/設計騙綠** |

**這套的精髓：把「口頭鐵則」升級成「機器會擋、會留痕」。** 不靠 AI 自律，靠工具強制：

- **客觀綠燈閘門 `gate.py`**：**判綠不由 PG 自述**。它客觀檢三條件（收集到的測試數 > 0、pytest 回傳碼 == 0、無 failed/errors），並比對契約檔的 sha256 沒被竄改——**只有它印出 `GREEN` 才算過**。這正面堵掉三個經典漏洞：collection error 被當「沒 failure＝綠」、`0 passed` 不算紅、PG 改測試騙綠。
- **角色硬隔離 hook `role_guard.py`**：一個 `PreToolUse(Write|Edit)` hook，讀目前角色（`.unet/role`），某角色想寫不屬於它的層、或想改基礎設施檔，直接 **`exit 2` 擋下**。把「prompt 自律」升級成「工具層攔截」。
- **契約不可下游偷改**：用 sha256 baseline 鎖死 `3_/4_`，PG 動到就 RED。
- **PG 的「停手出口」**：同一個失敗連修 2–3 次仍紅、或唯一弄綠的辦法是去動上游契約——**停手、回報、退一層**，不准就地竄改。每次退階都在 `ROADMAP.md` 留一行紀錄（日期／從哪退到哪／為什麼）。

**人機分工**
> **人介入的重點放在前三階段**（需求與設計）——因為「設計和驗收同時漏掉同一個概念」這種同源盲區，流程救不了，只能靠人攔。`/pm` 自己也誠實寫著「我和架構師是同一個模型，共享同一套盲區」，所以才強制它寫 metamorphic 測試、並用真實 E2E（不同 oracle）補獨立性。
> **`/pg` 才是放手讓它自主跑到全綠的地方。**

**一個值得學的態度：方法論自己也要誠實標註邊界。** 這套 skill 裡到處明寫「**這是縱深防禦，不是沙箱，別宣稱徹底解決**」「role_guard 擋得住 Write/Edit 越界，擋不住用 Bash 寫檔或把斷言邏輯搬進實作」「真正的 U-Net 只有三層，沒有 backprop」。把限制寫進工具本身，比假裝萬能可靠。

**什麼時候別用**：一次性腳本、拋棄式原型、單一純函式小工具——五層儀式的成本會超過直接寫。所以這套還內建 **Tier 分級**：無 I/O、無契約、無 GUI 的小模組走輕量路徑；一旦長出 I/O 或被別人依賴，**自動升級**走完整五層 + E2E。

## Tip 6 — persona + MCP 的真實 UX 驗收：「看截圖不算測試」

GUI 的驗收我堅持一條原則：

> **看截圖確認「畫面有載入」不算測試。真正的測試是「填真實資料 → 按按鈕 → 等非同步回應 → 驗證結果頁正確 → 切下一個任務確認狀態乾淨 → 測錯誤路徑」。**

`/ux-test` 的標準流程：
1. **Pre-flight**：確認 UI 進程、sidecar、每個外部依賴都活著（對每個受測模組 `curl` 它的 health-url，沒開就先啟動）。**任一失敗就停下來修好，不要跳過。**
2. **Golden Path**：逐模組「填真實資料 + 點按鈕 + 等回應 + 截圖驗證 + assert 關鍵文字」。填的是**真實值**（`operator001`）不是 placeholder。**第 4 步（真的去操作）是最重要、也最常被偷懶跳過的一步。**
3. **錯誤路徑**：把外部服務關掉，確認錯誤訊息是**友善的中文且有指引**（「無法連線至 AOI 系統」），不是噴原始 exception（`[WinError 10061]`）。
4. **persona 情境**：規劃 3 個不同背景的使用者（工廠操作員 vs 管理者 vs 工程師），找出「看起來正確但用起來混亂」的設計問題。

**把失敗「歸因」到正確的一層**（Tauri 版的 `e2e-test` skill 做得最徹底）：驗證的不是「element 存在」，而是 embedded Streamlit **真的在 WebView2 裡算繪出來**。而且 verdict 會分類：`PASS`（真的渲染）／`ENV_BLOCKED`（畫面 NOT_FOUND 但 `/_stcore/health` 回 200 → 是 engine/依賴問題，**與外殼無關**）／`INFRA`。最後明確聲明「這次失敗到底關不關外殼的事」——避免把鍋扣錯人。

**MCP 除錯的實戰眉角**（這些是踩出來的）：
- MCP **無法操作原生 `<select>`**，也點不了 canvas 格子 → 改用鍵盤或直接驗證後端邏輯。
- `st.error()` 在 `st.rerun()` 前呼叫會被清掉 → 錯誤訊息要存進 `session_state` 下次 render 再顯示。
- 工具剛 start 就截圖會 `ERR_CONNECTION_REFUSED`（Streamlit 要 2–3 秒起來）→ **輪詢 ready，不要盲目 sleep**。

## Tip 7 — False-Green 防身術（最重要的血淚教訓）

這是我吃過最多虧、也最值得分享的一條。

> **「pytest 全綠」≠「使用者真的能用」。「字串 grep 到了」≠「程式碼真的會跑」。**

真實踩過的坑（每一個都有畫面）：
- **測試只直呼 core registry，從未走活路徑** → scaffold 出來的 connector 灌進 core registry，但活的標注流程用的是另一個 labeling registry（不同工廠表），**永遠選不到**。測試全綠，功能 runtime 死。
- **external_gui 的 no-code 分支在檢查權限前提早 return** → 無 code 的外部 GUI 工具**繞過了 RBAC**。更狠的是：守門測試一開始是「字串位序」斷言（綠得很脆弱），後來升級成「用被拒權限的角色實際驅動，斷言啟動器永不被呼叫」才算數。
- **`enhance` 變數死在某個 `if shapes:` 分支內** → 沒有標注框的圖片根本不會走到它，所以「對比 toggle」切了沒反應。`grep enhance` 找得到、變數有讀，但**那條路徑不會跑**。
- **過寬的 `except Exception:` 吞掉了 Streamlit 的 `RerunException`** → `st.rerun()` 失效，按鈕「靜默無反應」。看起來是按鈕壞，其實是控制流的例外被吃了。
- **「修了沒生效」的元凶常常是：engine 還在跑舊碼**（改了沒重啟，`/reload` 不重載 `.py`），而你根本不知道跑的是哪一版。
- **空目錄 `glob` 靜默回 0 筆** → submodule 沒拉下來時，整列功能消失，但 `/health` 還是綠、app 照常啟動、**完全沒有錯誤**。

**因此固化下來的鐵則**：
1. **編輯後讀回確認**——平行做大量 `Edit` 時，個別 `old_string` 不匹配會**靜默失敗**，別假設 `Edit` 成功就是改對了。
2. **測語意 / 可執行性 / 活路徑**，不要測「字串存在」。涉及 GUI / 媒體 / 外部行為的，完成前**必跑真實 E2E**。
3. **宣稱完成前，跑完整測試套件**（`npm run test:python` + `npm test` + portal build），不要只跑改動的那幾個。
4. **讓「版本可見」**：啟動時印出 git commit hash + boot id（我埋的關鍵字是 `[CIM-BOOT]`），`/version`、`/diagnostics` 都能查——**一眼確認跑的是不是新碼**。

**meta 教訓**
> 我曾反覆「修了沒解決」，三個 agent 一起查出真正的根因不是某個 bug，而是：**沒有可重現的 PASS/FAIL 把關，全靠自我宣稱**（「看 DB 有沒有那筆」就說好了）。
> 驗證要「**實際跑啟動流程、攔截子程序的環境變數**」，而不是「看資料庫裡有沒有」。

## Tip 8 — 宣告式優先，用「確定性 scaffold」取代「AI 生成骨架」

一個反直覺、但很省事的習慣：**能不讓 AI 寫的 code，就不讓它寫。**

- 簡單工具改用 `plugin.yaml` 的宣告式 `form:` / `output:` / `external_gui:`，**零 Streamlit code**，框架自動渲染（連「啟動外部 GUI、回收結果、啟動前檢查 RBAC」都宣告式搞定，這就是 Label tool 模式）。
- 建工具骨架用**確定性的 CLI**（`python tools/scaffold.py module …`），CLAUDE.md 裡白紙黑字寫「**首選，免 AI agent**」。

**反面教材就在同一個 repo 裡**：早期共用碼靠 `importlib.util.spec_from_file_location` 用**字串路徑**動態載入——結果 IDE 的「Go to Definition」、`grep import`、靜態分析**全部抓不到**這些依賴，搬個檔案要到 runtime 才報 `FileNotFoundError`。這正是「動態、隱晦」的代價。

**為什麼值得抄**
> AI 生成的程式碼**每一行都是一個可能出錯的點**。宣告式 YAML 和確定性 scaffold 把「可能出錯的表面積」壓到最小——
> 該由人/框架決定的事（欄位、版面、骨架）用宣告式鎖死，**只把真正需要創造力的邏輯**交給 AI。

**同一個哲學還延伸到「事實來源」**：catalog 的唯一權威是宣告式 YAML，`tools.sqlite` 只是**衍生快取**，engine 每次啟動都從 YAML 冪等重建，所以**不進版控、不手動編輯**。凡是「可變、會漂移」的衍生物夾在 code 與畫面之間，就是 bug 的溫床（Tip 7 那個「修了沒生效」的根因正是如此）。這跟 Tip 7 是一體兩面：**減少 AI 與狀態能搞砸的地方，就是減少 false-green 的來源。**

## Tip 9 — 設定調校：模型、思考強度、權限白名單

幾個我長期累積的 `settings.json` 習慣：

- **`"model": "opus"` + 高 `effortLevel`**：開發這種需要深度推理、跨多檔案的工作，我固定用最強模型 + 最高思考強度。
- **逐步養大 permission allowlist**：把確定安全、會重複跑的命令（測試、特定腳本、唯讀查詢）加進 `allow`，減少每次都要按確認的打斷。
  - 提示：Claude Code 有 `/fewer-permission-prompts` 可以掃過往對話，自動幫你產出建議白名單。
- **`additionalDirectories`**：把 `~/.claude`（放共用指令/skill）也加進可存取目錄，跨專案共用。

**原則**
> 權限白名單是**用安全換流暢**。只加「唯讀」或「確定可逆、確定安全」的；破壞性命令（`reset --hard`、`push --force`、刪檔）**永遠保留人工確認**。

## Tip 10 — MCP + Log 分層除錯

GUI 出問題時，我有一套固定的分層查法，不亂猜：

1. **先用 MCP 截圖**確認「畫面現在長怎樣」（前端症狀）。
2. **再讀 Log 確認後端實際走了哪條路徑**——而且分清楚是哪一層：

| 層級 | Log 位置 |
|---|---|
| Streamlit input/output | `apps/host-electron/logs/streamlit-module_XXX-*.log` |
| Python 業務邏輯 | `tmp/cim_log/module_XXX_process.log` |
| FastAPI engine | `apps/host-electron/logs/engine.log` |

**為什麼值得抄**
> 「畫面壞了」可能壞在前端、橋接、engine、子程序、外部服務任何一層。
> **先定位是哪一層，再修**——比起對著一個症狀亂改，這能省下大量來回。

**而且我刻意把 log 設計成「方便貼給 AI」**：埋了統一關鍵字 `[CIM-PREFLIGHT]`（啟動前檢查、submodule 缺漏）、`[CIM-BOOT]`（commit hash + boot id），出事時直接搜、直接貼。這是「可觀測性 by design」——log 不只給人看，也是給 AI 的線索。
（一個冷知識級的坑：`.bat` 的 preflight 訊息**必須 ASCII-only**，因為下載者多是繁中（CP950）機器，UTF-8 中文 echo 不只會亂碼，還會讓 `cmd` 把訊息當指令執行。豐富的中文說明改放 UTF-8 的 `engine.log`。）

## 總結：這 10 個 tip 背後的 4 個心法

| 心法 | 對應 Tip |
|---|---|
| **給它記憶** — 讓 AI 跨 session 記住「看不出來但重要」的事 | 1、2 |
| **給它流程** — 把重複/關鍵的工作固化成指令、角色、評分機制 | 3、4、5 |
| **給它驗收** — 用「真實行為」而非「字串/截圖」證明事情真的成了 | 6、7 |
| **給它防呆** — 減少 AI 與狀態能搞砸的表面積，並用設定換流暢 | 8、9、10 |

還有一條貫穿全部的暗線——**對 AI 的產出保持誠實**：
- 分數不灌水（掉分照記、停在 94.4 就寫清楚為什麼不是 95）。
- 閘門會留痕（`gate.py` 印 `GREEN` 才算過、退階要在 `ROADMAP.md` 記一行）。
- 方法論自己標註邊界（「縱深防禦不是沙箱，別宣稱徹底解決」）。
- 「完成」的定義是真實行為跑通，不是自我宣稱。

**最後一句**
> AI 不會因為你「相信它」就做對。它會因為你**給它記得住的規則、跑得起來的驗收、和退得了的閘門**而做對。
> 把這些「協作基礎設施」建好，AI 的產出品質會穩定地往上走——這比換更強的模型，回報率更高。
